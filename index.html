关于createPackageContext

createPackageContext这个接口从名字可以猜测出来：根据传入包名创建该应用的上下文，在获得应用的上下文后，就可以进一步调用或者获取该应用的资源，甚至代码．当然，这取决于第二个参数：
CONTEXT_IGNORE_SECURITY：在创建上下文时候忽略任何安全限制，值是0x00000002
CONTEXT_INCLUDE_CODE：允许将被加载的应用的代码load到调用者的进程里执行，值是0x00000001
正如Google所警告的那样，同时使用CONTEXT_IGNORE_SECURITY和ONTEXT_INCLUDE_CODE是极其危险的．

基于createPackageContext的漏洞模式

根据createPackageContext的功能，我们可以得知：如果一个特权进程在调用createPackageContext时候，第一个参数，也即是应用包名是攻击者可以控制的，那么就有可能让特权进程加载恶意代码，从而实现提权．
包名可被攻击者控制有两种可能：
１）包名来自于对外暴露的组件
比如Activity,Service,ContentProvider,Receiver(包括动态注册的Receiver)
２）伪造包名
如果调用者在调用createPackageContext前，并没有校验被调用的应用是否存在，那么攻击者就可以伪造该应用．此外，即使使用了包名校验，但没有校验签名，那么对于非系统应用，攻击者可能会通过覆盖方式令调用者加载了恶意的应用代码．或者校验包名的时候，使用了不安全的校验方法，包括但不限于：
String.beginwith
String.contain
String.endwith
这样的例子，可以参考oversecured上的一个博客

关于createPackageContext的漏洞利用

一般情况下，调用者使用createPackageContext的顺序是：
１）调用createPackageContext,获得目标应用的上下文Context;
２）利用目标应用的上下文Context，调用getClassLoader;
３）根据getClassLoader,通过loadClass加载获得目标类对象
４）根据类对象调用getMethod获取方法/函数;
５）最后，调用invoke将代码load到调用者进程内执行
总的来说，就是通过反射方式将目标应用的代码load到调用者的进程上下文来执行．

在我们的常规理解里，调用者必须显式调用invoke来调用目标类的函数的时候，目标应用的代码才会加载到调用者进程的上下文，如果调用createPackageContext和getClassloader后必须在满足某些特定条件下才会执行后续的步骤，而这些特定条件可能是攻击者无法构造和满足的，这个情况下是否就无法利用了呢？
简而言之，就是仅调用了getClassLoader能否做得任意代码执行？
答案是肯定的．

１）在攻击APK的AndroidManifest.xml添加android:appComponentFactory属性

appComponentFactory属性的由来

从功能上来说，这个属性是为了应用在创建和初始化的时候，先调用appComponentFactory属性指定的类，这个类在实现的时候需要继承AppComponentFactory，一共６个接口：

关于CVE-2022-20421

CVE-2022-20421是最近一个关于Android Binder驱动的一个UAF漏洞，作者在OffensiveCon 23上进行了详细的分享,漏洞利用过程的详细介绍,同时，也公布了相应的EXP.这是一个非常复杂的漏洞利用，后续时间允许的话，再写一篇关于该漏洞的详细利用过程．

简单来说，这是一个关于队列自旋锁(queued spinlock)的UAF,触发场景是：进程A将一个指向进程C的handle分享给进程B的时候，如果进程Ｂ在该传输过程中退出，将会导致内核创建的binder_ref包含一个执行了已经退出的proc，而在进程C退出开始清理资源时候，访问了该proc的一个内部锁inner_lock,从而导致了UAF．

作者在他的漏洞利用过程中，使用多种技术，将队列自旋锁的UAF可以导致的修改内存最低两个字节为０的利用原语，转化为TTY write buffer的UAF,搭配pipe_buffer,实现了任意地址读写AAR/AAW,获得了手机的root权限．

selinux终结了root的上帝视角

在仅有DAC的时代，获得设备的root,也就意味着可以完全统治了该设备，直到MAC的出现．而现在手机默认开启了selinux这个MAC策略．获得了root后，能做的事情其实很有限，所以手机上的内核漏洞利用最后一步都需要想办法绕过selinux.

大部分内核漏洞利用中，在获取root权限的时候，通常都是利用AAR/AAW获取一个root进程的cred，然后将提权进程的cred修改为root进程的cred，比如init进程，该进程的pid总是１，很容易定位到该进程．

但是，目前init进程也不是万能的，尤其是该进程初始化的第２个阶段，会在应用selinux策略后，将自己的权限进行了下降，所以，即使获得了根init进程一样的权限，也无法绕过selinux.

绕过selinux

因此，绕过selinuxg成为内核漏洞利用过程中逃避的技术堡垒．不同的手机厂商对于selinux会开启不同的安全配置，甚至通过其它技术，比如TEE来增强selinux.
常见的安全配置包括：

  1. CONFIG_SECURITY_SELINUX_DISABLE
    如果配置了该编译参数，则意味者可以关闭selinux
  2. CONFIG_SECURITY_SELINUX_DEVELOP
    如果配置了该编译参数，则可以通过AAW将selinux_enforcing修改为０绕过selinux
    谷歌的pixel系列手机一般都会开启CONFIG_SECURITY_SELINUX_DEVELOP，所以，绕过谷歌pixel手机的selinux最容易．
    当然，一些手机由于谷歌的GKI要求，也会和谷歌保持一致，开启了CONFIG_SECURITY_SELINUX_DEVELOP．

以上命令不需要root权限．

多数手机厂商为了安全，一般都不会开启以上两个编译参数．在这种情况下，目前绕过selinux的方法有：

  1. 代码审计或者逆向，查看selinuxa权限检查过程中，是否存在可以关闭selinux的全局变量，如果存在，则可以通过AAW修改该全局变量的值来绕过selinux
    一般来说，权限检查都会调用到函数avc_has_perm,重点审计该函数．比如三星手机就有一个全局变量来控制–ss_initialized,通过将该变量修改为０，再利用libsepol API添加selinux的自定义策略就可以绕过selinux.详细技术细节可以参考作者的分享材料

    详细技术细节可以参考以下博客.但没有详细的exp代码
  3. 通过修改当前的selinux缓存策略，然后添加自定义策略

漏洞模式背景

JAVA是一个挺神奇的面向对象的编程语言，它提供了一个反射机制，通过该机制，可以：
1、获取一个指定的Class对象
2、获取指定类的所有字段–Field
3、获取指定类的所有构造方法 –constructor
4、获取指定类的所有非构造方法 –Method
这个特性原意可能仅仅是辅助开发人员有效地开发和自验证，从安全角度看，却提供了一个很好的漏洞利用方法。

漏洞模式

在Android中，如果应用程序，包括框架层，JAVA代码中如果使用了static定义成员变量，那么该变量的值总是可以被第三方应用获取，甚至修改，如果没有设置final属性的话。
遇到过以下的几种情形：
1、使用static变量存储密钥
2、使用static变量存储白名单
3、使用static变量存储路径
对于第一种，显然会导致敏感信息泄漏，对于第二和第三中情形，如果开发人员忘记设置final属性，那么攻击者可以通过反射机制修改，从而绕过白名单校验，路径校验等，严重的话甚至可以导致任意代码执行
如何搜索该类漏洞：
可以通过反编译工具，比如jadx，jeb等对APK或者JAR包进行反编译，然后根据以下正则表达式进行搜索：
^((?!final).)static((?!final).)$
以上正则表达式的含义是：搜索包含static但不包含final的字符串，原因是包含final关键字的字段是无法修改的。

如何利用漏洞

通常情况下，可以直接使用Class.forName获取相应的Class对象，然后通过该Class对象的getDeclaredField/getDeclaredMethod等方法获取变量属性和成员方法，具体例子可以参考链接

如果不是在该环境变量下的JAR包，那么可以这样操作：
1）在攻击APK的manifest.xml文件里添加queries属性，值是被攻击的包名

3）调用Class.forName(“目标类名”，false，otherappcontext.getClassLoader())

Android中的任意代码执行漏洞

发表于 2023-08-01 分类于 漏洞模式
本文字数： 1.6k 阅读时长 ≈ 1 分钟

关于任意代码执行漏洞

与任意命令执行漏洞类似，但任意代码执行不仅仅可以执行系统中已经存在的命令，还可以执行攻击者指定的代码（这通常通过执行一段shellcode，或者加载执行攻击者编写的一个二进制文件，比如可执行文件，动态库等）。
在Android中，也会有类似的漏洞。

Android中的任意代码执行漏洞

1、应用程序加载的动态库可被攻击者控制
应用程序基于各种业务需求，通常会将一些公共功能，尤其是涉及性能方面的，在一个动态库so里实现，比较，文件解析，媒体编解码，通信，加解密。这种情况下，通常是使用了JNI技术，最后通过System.load/System.loadLibrary加载相应的动态库。
其中，System.load输入的是动态库的绝对路径，可以是任意路径（当然前提是该应用有访问权限）;而System.loadLibrary的参数是动态库的文件名，不包括扩展名，比如动态库libtest.so,则只需调用System.loadLibrary(“test”)即可。
对于System.loadLibrary稍微展开一下：由于不是路径，而只是一个动态库文件名，那么必然会涉及文件搜索的过程，是否会存在Window平台下的动态库劫持问题呢？
System.loadLibrary(String libName) –> Runtime.getRuntime().loadLibrary(libName, VMStack.getCallingClassLoader())
这里因为涉及比较复杂的代码分析，因此仅是总结以下：默认先从应用安装目录下lib文件夹搜索，如果没找到，就到一个系统变量：java.library.path中查找，一般情况下，该系统变量值是：/system/lib和/vendor/lib。因此要实现动态库劫持，最佳方法是利用其它漏洞篡改应用安装目录下的动态库

2、DexClassLoader路径可被攻击者控制
DexClassLoader是用于加载外部的类，比如一些插件，是动态从网络的服务器下载然后load到应用的进程空间。
DexClassLoader(String dexPath, String optimizedDirectory, String librarySearchPath, ClassLoader parent)
如果第一个参数dexPath是攻击者可以控制的，那么就可以令应用程序加载攻击者指定的dex，实现任意代码执行。
第二个参数是一个缓存目录，一般是沙箱内的私有目录
第三个是dex可能需要依赖so搜索路径
第四个参数是父类加载器

3、createPackageContext的包名可被攻击者控制或者绕过
应用程序可能需要获取其它应用的接口，包括Java的方法，应用的资源等。由于其它应用Java的方法是不能直接loadclass的，因此其它应用的classloader与当期应用的classloader不一样，因此需要先通过createaPackgeContext将应用load到内存，获得context后，再getClassloder，最后通过loader来loadClass。这里可能存在几个可能的漏洞模式：
1）createPackageContext第一个参数，也即是packageName是攻击者完全可控
攻击者可以伪造按照该packageName的应用，被动等待，一旦受害者应用调用了该接口，哪怕仅是调用getClassloader都可以触发任意代码执行：

2）packageName的校验可被绕过
比如使用startWith，endWith，contain，indexof，这些方法很容易被绕过
具体的漏洞模式解释，可以参考这个博客

关于mariana trench静态分析工具

mariana trench是facebook，也即是现在的Meta公司开源的一款用于Android的静态分析工具，支持有源码和无源码两种方式的分析，详细使用可以看官网的介绍,源码在github上开源.
其它同类的工具还有Mobile-Security-Framework-MobSF、Soot、appshark、Jandroid。
mariana trench在有源码的情况下，会比较明显指出漏洞所在的代码，无源码的情况下，通常需要更多的人工分析确认。与Soot、appshark、Jandroid一样，mariana trench最大的特点就是静态污点分析，通过预置和自定义的规则查找满足一定模式的漏洞。
经过一段时间的使用，mariana trench有3个比较明显的缺陷：
1）主要是基于源码（反编译后的）进行污点分析，但是对于漏洞模式是出现在manifest配置的漏洞则无法制定相应的规则;
2）与其它几款污点分析工具类似，也不支持对jar包进行分析，需要一些工具将jar转化为apk;
3）默认情况下，不支持对批量的apk进行分析，需要自行编写脚本进行批量分析apk。
此外，学习曲线也比较陡，需要充分理解规则里的各个字段的含义，否在即使知道了漏洞的模式，也不知道该怎么写规则。

不恰当动态授权grantUriPermission导致的漏洞

Android的应用沙盒

在Android操作系统中，每一个应用都会在安装的时候分配一个独一无二的用户ID–UID，然后每个应用都会在各自的进程中运行。Android利用分配给应用的UID设置一个内核级别的应用沙盒（所以，如果能获得内核权限，应用沙盒机制自然就失效了），内核利用Linux安全机制（比如DAC，MAC）对应用和系统之间进行保护，默认应用之间不能相互访问。
详细可以看Android官网上的介绍

Android应用之间的文件共享

但，毫无疑问，应用之间必然存在文件共享访问的需求。比如A应用请求图库打开一张图片，请求播放器播放一个视频文件等等。在这种情况下，授权应用需要将文件–通常对应一个Uri的读权限或者写权限授予被授权的应用。根据最小授权的原则，授权应用应该仅授予目标应用相应的权限（读或者写），并且在事务完成后取消授权。
因此，Android在安全最佳实践中，提供的建议如下:

原因是通过Intent的addFlag设置的读写权限属于一次性授权，事务完成后授权自动取消，从而保证了最小授权。

grantUriPermission漏洞模式

与使用Intent的addFlags一次性授权相对应的是使用grantUriPermission函数进行持久授权

第一个参数是被授权的应用的包名，第二个参数是被授权的文件的Uri，第三个参数是具体的授权方式：读，写，同时同时具有读写权限。
从函数的参数可以看到，如果攻击者可以控制第一个参数，那么就可以获得了文件的读写权限。这可能包括以下几种情况：
1）包名是从外部可被攻击者控制的输入传入，比如导出的组件（四大组件），webView等
进一步可能存在两种情况：1是包名完全从外部可控输入，2是包名的不完整校验导致绕过。比如使用startWith，endWith，contain等不安全的字符校验。
2）授权的包名当前系统中不存在，因此攻击者可以伪造该包名，从而获得文件的读写权限。
3）最常见的一种，就是通过隐式Intent查询当前系统满足能处理该Intent的包，然后授予这些应用授权。

这种授权方式，明显属于授权范围过大。攻击者只需要定义一个Activity，然后Intent-filter设置Action是android.intent.action.SEND就可以获得文件的读写权限。
虽然，这种情况下，通常需要用户交互，而且Uri获取的确困难，因此CVSS打分通常无法达到7.0以上，仅属于一般问题。但是，如果Uri是有规律，可预测的（哪怕是遍历沙盒的文件），CVSS打分也将达到7.0以上，属于严重安全问题。

总结

对于Android开发而言，不建议使用grantUriPermission进行授权，推荐使用Intent的addFlags进行一次性授权，防止出现授权过大的安全风险。
